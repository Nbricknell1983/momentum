I use replit to build code, firebase as a datbase. how would the dev team build new code     
                                                                                              

 If Momentum uses Replit for build and Firebase as the database/backend platform, then the    
 dev team should build new code with a controlled workflow that separates planning,           
 implementation, validation, and release.                                                     
                                                                                              
 Recommended ownership                                                                        
                                                                                              
 - Strategy / Client Strategist / Operations                                                  
 define objective, constraints, and priority                                                  
 - Full Stack Developer                                                                       
 lead end-to-end implementation                                                               
 - Frontend Developer                                                                         
 own UI, responsiveness, interaction quality                                                  
 - Backend Developer                                                                          
 own Firebase data model, auth, functions, integrations, reliability                          
 - CRM & Automation Engineer                                                                  
 own workflow/event integrations if the feature touches lead routing, messaging, calendars,   
 or automation                                                                                
 - QA / Tester                                                                                
 validate journeys, regressions, edge cases                                                   
 - DevOps / Systems Engineer                                                                  
 own environment setup, deployment flow, secrets, release safety                              
                                                                                              
 How the dev team should build new code                                                       
                                                                                              
 ### 1. Define the change before touching code                                                
                                                                                              
 For every new feature or change, start with a short implementation brief:                    
                                                                                              
 - problem                                                                                    
 - desired outcome                                                                            
 - owner                                                                                      
 - user flow                                                                                  
 - data affected                                                                              
 - systems affected                                                                           
 - risks                                                                                      
 - success criteria                                                                           
                                                                                              
 Example:                                                                                     
 - Problem: leads are captured but not routed reliably                                        
 - Outcome: every lead is assigned within 30 seconds with source + owner logged               
 - Systems: Replit app, Firebase Firestore, auth, Twilio notifications                        
 - Risks: broken lead flow, duplicate records, cross-client leakage                           
 - Success: 100% of test leads routed correctly                                               
                                                                                              
 This prevents “build first, clarify later.”                                                  
                                                                                              
 ────────────────────────────────────────────────────────────────────────────────             
                                                                                              
 ### 2. Decide what belongs where                                                             
                                                                                              
 With Replit + Firebase, the team should be explicit about responsibility boundaries.         
                                                                                              
 Frontend                                                                                     
 - pages                                                                                      
 - forms                                                                                      
 - dashboards                                                                                 
 - client-side validation                                                                     
 - user interaction/state                                                                     
 - display logic                                                                              
                                                                                              
 Firebase                                                                                     
 - Firestore data model                                                                       
 - Auth                                                                                       
 - Cloud Functions / backend logic                                                            
 - role and permission logic                                                                  
 - triggers / automations                                                                     
 - audit/event logging                                                                        
                                                                                              
 Rule:                                                                                        
 Do not put sensitive business logic only in the frontend if it should be enforced            
 server-side.                                                                                 
                                                                                              
 If it affects:                                                                               
 - permissions                                                                                
 - billing                                                                                    
 - tenant boundaries                                                                          
 - lead routing                                                                               
 - workflow control                                                                           
 - critical calculations                                                                      
                                                                                              
 …it should be enforced in backend logic / Firebase rules / functions, not just UI code.      
                                                                                              
 ────────────────────────────────────────────────────────────────────────────────             
                                                                                              
 ### 3. Build in a branch or isolated copy, not directly in the live app                      
                                                                                              
 Even in Replit, the team should avoid cowboy edits in production.                            
                                                                                              
 Preferred pattern:                                                                           
 - create a feature branch or isolated development repl/environment                           
 - implement there                                                                            
 - test there                                                                                 
 - merge/deploy only after validation                                                         
                                                                                              
 This matters because Replit makes it easy to move fast, but fast without isolation creates   
 regression risk.                                                                             
                                                                                              
 ────────────────────────────────────────────────────────────────────────────────             
                                                                                              
 ### 4. Design the Firebase layer first for any meaningful feature                            
                                                                                              
 Before building screens, the Backend Developer / Full Stack Developer should define:         
                                                                                              
 - collections / documents                                                                    
 - tenant/client separation model                                                             
 - auth roles                                                                                 
 - read/write permissions                                                                     
 - required indexes                                                                           
 - event triggers                                                                             
 - audit trail requirements                                                                   
                                                                                              
 Questions they should answer:                                                                
 - where does this data live?                                                                 
 - who can read it?                                                                           
 - who can write it?                                                                          
 - how is tenant isolation enforced?                                                          
 - what happens if a write fails?                                                             
 - what downstream workflow depends on this data?                                             
                                                                                              
 If you skip this and build UI first, you usually end up rewriting logic later.               
                                                                                              
 ────────────────────────────────────────────────────────────────────────────────             
                                                                                              
 ### 5. Implement smallest safe vertical slice first                                          
                                                                                              
 Do not try to ship the entire feature in one pass.                                           
                                                                                              
 Build the minimum end-to-end slice:                                                          
 - UI input                                                                                   
 - backend write                                                                              
 - validation                                                                                 
 - confirmation/output                                                                        
 - logging                                                                                    
 - QA pass                                                                                    
                                                                                              
 Example:                                                                                     
 If building a new lead intake workflow, first ship:                                          
 - one form                                                                                   
 - one Firestore write path                                                                   
 - one assignment rule                                                                        
 - one confirmation state                                                                     
 - one test notification                                                                      
                                                                                              
 Then expand.                                                                                 
                                                                                              
 That gives:                                                                                  
 - proof the architecture works                                                               
 - early QA                                                                                   
 - lower regression risk                                                                      
 - faster learning                                                                            
                                                                                              
 ────────────────────────────────────────────────────────────────────────────────             
                                                                                              
 ### 6. Use Firebase safely                                                                   
                                                                                              
 For Firebase specifically, the dev team should enforce these controls:                       
                                                                                              
 Auth                                                                                         
 - role-based access                                                                          
 - no overbroad permissions                                                                   
 - secure token/session handling                                                              
                                                                                              
 Firestore / DB                                                                               
 - clear schema conventions                                                                   
 - tenant/client IDs on all relevant records                                                  
 - security rules reviewed before release                                                     
 - indexes planned for query-heavy features                                                   
                                                                                              
 Cloud Functions / backend logic                                                              
 - move critical workflow logic server-side                                                   
 - validate all inputs                                                                        
 - log failures                                                                               
 - handle retries/idempotency where needed                                                    
                                                                                              
 Storage / files                                                                              
 - scoped access rules                                                                        
 - no public exposure by accident                                                             
                                                                                              
 Non-negotiable:                                                                              
 Tenant isolation and permission boundaries need to be designed, not assumed.                 
                                                                                              
 ────────────────────────────────────────────────────────────────────────────────             
                                                                                              
 ### 7. QA before release                                                                     
                                                                                              
 QA / Tester should validate:                                                                 
                                                                                              
 Core journey                                                                                 
 - happy path works end-to-end                                                                
                                                                                              
 Regression checks                                                                            
 - existing flows still work                                                                  
                                                                                              
 Edge cases                                                                                   
 - missing data                                                                               
 - duplicate submits                                                                          
 - auth expiry                                                                                
 - wrong permissions                                                                          
 - slow network / retries                                                                     
 - partial failure states                                                                     
                                                                                              
 Workflow validation                                                                          
 - are automations triggered correctly?                                                       
 - are logs written?                                                                          
 - are notifications duplicated or missing?                                                   
 - does routing go to the right place?                                                        
 - does any client see another client’s data?                                                 
                                                                                              
 For your stack, QA should especially focus on:                                               
 - Firebase rules                                                                             
 - write/read permissions                                                                     
 - multi-tenant safety                                                                        
 - triggered workflow reliability                                                             
                                                                                              
 ────────────────────────────────────────────────────────────────────────────────             
                                                                                              
 ### 8. Release with rollback thinking                                                        
                                                                                              
 DevOps / Systems should make sure release is controlled:                                     
                                                                                              
 - environment variables/secrets correct                                                      
 - Firebase config matched to correct project                                                 
 - production deployment checklist exists                                                     
 - rollback path exists                                                                       
 - monitoring/logging is visible after release                                                
                                                                                              
 A common failure mode here is:                                                               
 - code points to wrong Firebase project                                                      
 - test config leaks into prod                                                                
 - rules deploy incorrectly                                                                   
 - automations fire in the wrong environment                                                  
                                                                                              
 So release should include:                                                                   
 - preflight check                                                                            
 - deployment                                                                                 
 - smoke test                                                                                 
 - post-release verification                                                                  
                                                                                              
 ────────────────────────────────────────────────────────────────────────────────             
                                                                                              
 ### 9. Verify outcome, not just deployment                                                   
                                                                                              
 After shipping, Bullpen’s standard applies:                                                  
                                                                                              
 Verify:                                                                                      
 - did the feature actually solve the problem?                                                
 - is quality acceptable?                                                                     
 - did any regressions appear?                                                                
 - is the business outcome moving?                                                            
                                                                                              
 Measure against the original success criteria:                                               
 - time saved                                                                                 
 - conversion improved                                                                        
 - routing accuracy                                                                           
 - fewer support issues                                                                       
 - better visibility                                                                          
 - higher reliability                                                                         
                                                                                              
 ────────────────────────────────────────────────────────────────────────────────             
                                                                                              
 Recommended delivery workflow                                                                
                                                                                              
 For new code, the dev team should usually work in this order:                                
                                                                                              
 1. define the problem and outcome                                                            
 2. assign lead owner                                                                         
 3. map impacted systems                                                                      
 4. design Firebase data/auth/rules structure                                                 
 5. build smallest safe end-to-end slice in Replit                                            
 6. run QA on journey + regressions + permissions                                             
 7. deploy with environment/release controls                                                  
 8. verify production behaviour                                                               
 9. feed learning into the next iteration                                                     
                                                                                              
 ────────────────────────────────────────────────────────────────────────────────

 - UI work

 - integration testing                                                                        
 - rapid implementation                                                                       
                                                                                              
 Firebase                                                                                     
 - Firestore database

 - logs/events for critical backend actions                                                   
                                                                                              
 Rule of thumb                                                                                
 - Replit = development environment and app layer                                             
 - Firebase = state, backend services, auth, and protected business logic
