Build Brief — OpenClaw Task Submission API Feasibility & Integration Assessment
Goal
Determine whether Bullpen can dispatch work to actual OpenClaw runtime agents through a supported task submission API, and if so, define the integration contract clearly enough to implement real runtime workforce execution.

Problem
Bullpen now has real two-stage specialist routing, but it is still implemented entirely through OpenAI API calls inside one orchestration layer.

Current state
Stage 1: routing decision via OpenAI
Stage 2: specialist-context prompt via OpenAI
output quality differs by specialist context
no actual task is being sent to named OpenClaw runtime agents
What is missing
For Bullpen to truly dispatch work to real OpenClaw agents like:

frontend-developer
backend-developer
qa-tester
operations-specialist
Bullpen would need a supported way to:

target a specific OpenClaw agent ID
submit a task payload
receive a result or completion signal
track status
The core uncertainty
Current OpenClaw integration work confirmed:

gateway/dashboard contract exists
health/identity verification works
provisioning path is gateway-oriented
But it is not yet confirmed that OpenClaw exposes a task submission API Bullpen can POST to.

Without that, real runtime dispatch is not implementable.

Outcome that matters most
Get a definitive answer on whether Bullpen can use actual OpenClaw runtime agents as execution targets.

Success means one of two outcomes:

Outcome A — supported
OpenClaw exposes a real task submission contract, and you can define:

endpoint(s)
auth
payload shape
targeting by agent ID
sync/async behavior
status/result handling
Outcome B — not supported
OpenClaw does not currently expose that contract, and the product/docs/UI should continue using specialist-context orchestration instead of pretending runtime dispatch exists.

Both outcomes are useful, as long as they are definitive.

Scope
In scope
inspect OpenClaw docs/runtime for task submission capability
determine whether specific agent IDs can be targeted programmatically
define request/response contract if supported
identify sync vs async execution model
identify auth and gateway requirements
recommend next implementation step based on findings
Out of scope
building a fake dispatch layer
pretending existing gateway health checks imply task execution support
broad OpenClaw redesign
unrelated Bullpen UI work
Required architecture rules
1. Do not imply runtime dispatch without a real contract
No UI copy or implementation should claim “dispatched to real OpenClaw agent” unless the API path actually exists.

2. Health verification is not task execution
Just because Bullpen can verify the OpenClaw gateway does not mean it can submit work to agents.

3. Agent config existence is not dispatch support
Just because agents are listed in OpenClaw config/runtime does not mean Bullpen can call them programmatically.

4. Produce a binary answer
This assessment should end in either:

yes, here is the contract
or
no, this is not currently supported / not yet discoverable
