Build Brief — Proactive Prep Readiness for All Active Leads
Goal
Ensure every active lead has a strong, current Prep Call Pack and hydrated Deal Intelligence before Nathan opens the record, so the agents are doing the preparation work in advance rather than reacting when the lead is clicked.

Core outcome
The agents should walk into the lead before Nathan does.

That means:

prep is generated proactively
deal intelligence is already hydrated
stale prep is refreshed automatically
priority leads are ready first
opening a lead reveals a prepared commercial point of view, not a record that is still waiting to think
Problem
The latest build improved prep quality and hydrates deal intelligence, but prep generation is still too reactive:

it triggers on lead open
it depends on Nathan touching the record first
it does not yet guarantee that active leads are already prepared in advance
That means the agents are still:

helping at the moment of interaction
instead of:

doing the work before interaction
If these are the workforce, they should already have prepped the important leads before Nathan arrives.

Outcome that matters most
Make prep readiness a background operating behavior for all active leads.

Success means:

active leads are automatically prepped on a cadence
stale prep is refreshed before it becomes a problem
deal intelligence is already hydrated when Nathan opens the card
the highest-priority leads are always prepared first
manual prep generation becomes an exception, not the default path
Scope
In scope
scheduled prep generation for all active leads
lead prioritisation for prep ordering
freshness logic for prep packs
automatic deal intelligence hydration from prep results
visible prep-ready state on lead records/list if useful
safeguards against unnecessary reprocessing
Out of scope
prep generation for won/lost leads
broad client strategy refresh in this task
redesign of the prep pack content model
external notifications
full call scheduling integration unless already trivial
Required architecture rules
1. Prep should happen before the click
Opening a lead should reveal prepared intelligence, not start the thinking process.

2. All active leads should be eligible
The system should process all active leads eventually, not only whichever one Nathan opens first.

3. Priority-first ordering
If resources are constrained, the system should prep the highest-value leads first.

4. Freshness should be explicit
Prep should not rerun blindly.
Use freshness windows and change-awareness where possible.

5. Deal intelligence hydration must stay automatic
When prep updates, the deal intelligence layer should stay in sync automatically.

Recommended implementation approach
Phase 1 — Scheduled prep job for active leads
Add a scheduled prep readiness job that runs independently of record open.

Scope
all active leads
exclude won/lost/archived where appropriate
Recommended cadence
at least once daily
optional additional pass during business hours later if useful
Goal
Every day, the system should work through active leads and bring prep up to date.

Phase 2 — Lead prioritisation for prep order
Do not process leads in a random or opaque order.

Recommended prep priority order
Use a weighted ordering such as:

leads with upcoming or recent activity / call relevance
leads in active pipeline stages
leads with no prep pack yet
leads with stale prep
leads recently updated
older untouched active leads
Goal
The most commercially relevant leads should be prepared first.

Phase 3 — Freshness logic
Define when a prep pack is considered stale.

Suggested rules
Refresh prep if:

no prep exists
prep older than threshold
lead data changed materially
website / presence / enrichment changed materially
stage/progression changed materially
manual force refresh requested
Recommendation
Move beyond simple “older than X days” where practical.
If change detection is easy, use it.

Phase 4 — Background hydration of Deal Intelligence
When prep is generated by the scheduled process, the system should also hydrate:

AI Deal Summary
Agent Intelligence card
Strategy Intelligence empty fields where allowed
That way, when Nathan opens the lead later, the intelligence is already there.

Goal
No user-triggered gap between prep generation and visible deal intelligence.

Phase 5 — Prep readiness visibility
Show a visible state that tells Nathan whether a lead is actually prepped.

Example states
Ready
Refreshing
Stale
Not prepared
Failed / blocked
Possible locations
lead list/table badge
deal card header
prep tab header
Goal
Nathan should know whether the lead has been prepped without guessing.

Phase 6 — Batch safety / queueing
If lead volume is large, process in batches safely.

Requirements
avoid hammering OpenAI/API limits
avoid reprocessing fresh leads unnecessarily
record progress
resume gracefully if interrupted
Goal
Prep readiness should scale without becoming noisy or expensive.

Risks and dependencies
Risks
repeatedly regenerating prep that hasn’t materially changed
processing lower-value leads before hot ones
showing “ready” when the prep is actually stale
spending too much AI budget on low-value refreshes
Dependencies
current prep pack engine
current enrichment signals
current deal intelligence hydration behavior
any lead activity/stage metadata available for prioritisation
QA requirements

Must test — priority

Must test — freshness

Must test — hydration

manual Strategy Intelligence fields are still respected if already filled
