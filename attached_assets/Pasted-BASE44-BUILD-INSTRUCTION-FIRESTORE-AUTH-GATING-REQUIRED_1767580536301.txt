1) FIX AUTH + MEMBERSHIP VERIFICATION GATING (CRITICAL)
Problem

Queries are being started before membership verification completes, causing:

“BLOCKED: READ - authReady is false”

“Auth not ready - membership not verified”

Requirement

Implement a strict 2-stage readiness gate:

authReady: Firebase Auth state is known (user is signed in)

membershipReady: membership doc exists and active == true

Membership source of truth

Membership is defined by this document existing and being active:

/orgs/{orgId}/members/{uid} with field active == true

New rule

Do not mount/subscribe/query any org collections until:

authReady === true

orgId is resolved

membershipReady === true

If membership is not ready:

show a blocking UI state (“Verifying access…”) and do not start any Firestore subscriptions.

Implementation behavior

onAuthStateChanged fires

load /users/{uid} to resolve orgId

read /orgs/{orgId}/members/{uid}

if exists AND active=true → set membershipReady = true

only then start:

leads, clients, activities

tasks, dailyPlans

aiBriefs, aiDebriefs

actionRecommendations

2) FIRESTORE COLLECTIONS REQUIRED (UNDER orgs/{orgId})

3) REQUIRED COMPOSITE INDEXES (FIX “QUERY REQUIRES AN INDEX”)

Base44 must ensure Firestore composite indexes exist for the exact queries the app is running.

3.1 tasks composite index

For Daily Plan and ordered chips, the app queries tasks by:

planDate

userId
and orders by:

sortOrder
(plus name)

Create this composite index:

Collection Group: tasks
Fields:

planDate Asc

userId Asc

sortOrder Asc

__name__ Asc

3.2 actionRecommendations composite index

For AI action queue recommendations, the app queries by:

planDate

userId
and orders by:

priorityScore (descending)
(plus name)

Create this composite index:

Collection Group: actionRecommendations
Fields:

planDate Asc

userId Asc

priorityScore Desc

__name__ Desc (or Asc if Base44 requires; match Firestore’s generated index)

4) DATE FORMAT REQUIREMENT (DD-MM-YYYY)
User-facing date format

All displayed and stored plan date strings must use:

DD-MM-YYYY (example: 05-01-2026)

This includes:

Daily Plan selected day

Task planDate

Daily Plan document IDs (if used)

AI Brief/Debrief doc IDs (if used)

Internal sorting (important)

Because DD-MM-YYYY is not lexicographically sortable across months/years, the system must also store:

planDateKey = YYYY-MM-DD (internal only, not shown in UI)

This field is purely for correct week/month sorting and range queries.

Non-negotiable:

UI shows DD-MM-YYYY only

Firestore can store both planDate (DD-MM-YYYY) and planDateKey (YYYY-MM-DD internal)
